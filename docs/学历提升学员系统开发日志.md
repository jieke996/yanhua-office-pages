## 2026-06-23 最高管理员 admin 账号排查

### 需求或问题背景

用户确认存在一个 `admin` 账号作为最高管理员，但线上只读接口 `/api/admin/users` 未返回该账号，因此需要排查 `admin` 是否写死在前端或后端逻辑中。

### 本次关键操作行为

- 搜索 `index.html`、`home.html`、`admin.html`、`config.js` 和 `tools/` 下关于 `admin`、`currentUser`、`localStorage`、`login`、`role`、`api/admin` 的实现。
- 读取 `index.html` 登录逻辑，确认前端通过 `/api/auth/login` 登录，成功后把后端返回的 `result.user` 写入 `localStorage.yanhua_user`。
- 读取 `home.html` 首页逻辑，确认后台入口显示条件是 `currentUser.role === 'admin'`。
- 读取 `admin.html` 后台逻辑，确认后台页访问条件也是 `currentUser.role === 'admin'`。
- 复查 `/api/admin/users` 只读返回结果，当前仍只返回“李佳新”“孟祥琳”两个普通用户账号，不返回 `admin`。

### 排查结论

当前前端代码未发现写死 `admin` 账号密码，也未发现前端本地创建 `admin` 账号。前端只是根据后端登录接口返回的用户对象判断角色：

- `result.user` 保存到 `localStorage.yanhua_user`。
- `currentUser.role === 'admin'` 时显示后台管理入口。
- `currentUser.role === 'admin'` 时允许进入后台管理页。

因此，`admin` 账号更可能是在后端认证逻辑、后端默认账号、环境变量、配置文件或后端数据库的特殊记录中处理；也可能被 `/api/admin/users` 接口刻意排除，避免在员工管理列表中显示最高管理员。

### 是否影响数据库、上传文件、配置文件、线上业务数据

本次只读排查，不修改数据库、上传文件、配置文件和线上业务数据。

### 后续建议

- 学历提升学员系统第一版可以把 `role === 'admin'` 作为最高管理员识别条件，与现有后台管理保持一致。
- 后端实现时仍必须在接口层校验 `role === 'admin'`，不能只依赖前端隐藏入口。
- 如果后续要区分“普通管理员”和“最高管理员”，再引入 `super_admin` 或 `permissions.educationStudentManage`。
